AD ALERT DOCUMENTATION

AdAlert（アドアラート）機能ガイド
― 検知アラート仕様書

このドキュメントは、AdAlert が広告配信を どう監視し、どんな時にアラートを出すか を、広告運用の担当者が読んで分かる形でまとめたものです。各アラートの「どういう時に鳴るか」「鳴ったらどうするか」を中心に記載しています。

この資料での用語

  • 広告表示＝インプレッション（広告が表示された回数）。画面上の名称に合わせています。
  • 利用額＝コスト（消化金額）。
  • 配信予定（計画）＝AdAlert 上で設定した配信開始/停止の予定。媒体側の実際の設定とは別に、AdAlert が「正」として持つ計画です。
  • 案件＝複数の広告アカウントを媒体をまたいでまとめた予算管理の単位（fee率・月予算／期間予算を持つ）。

重要度の見方

  • 重大　… 配信事故につながる、すぐ確認すべきもの（画面では赤）
  • 注意　… 想定外の動きなので確認したいもの（画面では黄）

ランクの見方（対応の優先度）

アラートは件数が多く、「対応済みにする作業」そのものが現場の負担になりがちです。そこで各アラートに、対応の優先度＝ランク（3段階） を自動で付け、「緊急は1件ずつ／誤検知はまとめて対応済み」 と作業を振り分けられるようにしました。

重要度（重大／注意）が「変化の大きさ」を表す従来の区分なのに対し、ランクは「今すぐ対応すべきか」を表す優先度です。画面のアラートにはランクが表示されます（例：広告表示が1→5に増えると倍率上は大きく＝重要度は上がりますが、実害が無ければランクは低くなります）。

ランク1 緊急放置すると損害が出る／出るべき広告が止まっている。1件ずつ確認して対応します（まとめて閉じない）。
ランク2 要確認問題の可能性はあるが、確実な損害とは言い切れないグレー。個別に確認します。
ランク3 誤検知濃厚検知はされたがデータ的に問題なし（小さな数字・ロジック上のノイズ）。まとめて対応済みにできます。

※ 数値の目安（利用額1,000円・表示100回・実績1,000回／10回 など）は初期値で、「監視・アラート設定」画面から会社ごとに変更できます（変更は既存アラートのランクにも即反映されます）。

ランク判定の詳しい条件（数値ルール）

各アラートが 1／2／3 のどれになるか、具体的な条件は次のとおりです。下記の数値は「監視・アラート設定」画面から会社ごとに変更できます（初期値を記載）。

新規未表示（ゼロ表示）
①復帰チェック：検知の後 6時間以内に表示が10回以上 出ていれば「配信が戻った（一過性）」とみなし 誤検知(3)。
②戻っていなければ、検知直前7日間の表示実績で判定 ―― 1,000回以上 → 緊急(1)／10回以下 → 誤検知(3)／その間 → 要確認(2)。
補足：表示が1回ポツンと出た程度は「戻った」に数えません（6時間で10回以上が必要）。また6時間より後に戻った場合（例：9時間後）は今の設定では拾わず緊急のまま残ります。実績データが未取り込みのときはランクを付けません。
停止後表示
停止後に続く表示が 10回以上 → 緊急(1)／10回未満 → 要確認(2)。実損に直結する型なので誤検知(3)には落としません。
効果大幅変動（広告表示）
前日・当日の多い方が 100回未満 → 誤検知(3)／100回以上 → 要確認(2)。表示回数の増減はお金が直接動かないため、緊急(1)にはしません。
効果大幅変動（利用額）
増加 かつ 当日1,000円以上 → 緊急(1)（予算が溶けている恐れ）／前日・当日とも 1,000円未満 → 誤検知(3)／それ以外（減少・中規模） → 要確認(2)。
利用率異常（予算着地）
着地予測が 大幅な逸脱（90%未満 または 110%超） → 緊急(1)／軽度の逸脱（90〜97% または 99〜110%） → 要確認(2)。目標範囲は案件・会社ごとに変更できます。
決済エラー
常に 緊急(1)。広告アカウント全体（配下の全配信）に波及するため、警告段階でも緊急として扱います。
ランク対象外
広告審査落ち・CV未発生・予算変更スパイク はランクを付けません（対象外）。審査落ちは承認復帰で自動解決、予算変更スパイクは意図確認が本質、CV未発生は今後ランク付けを追加予定です。

使い方（トリアージ）

  • ランク3（誤検知濃厚）はまとめて対応済み：受信箱と 効果大幅変動・新規未表示（ゼロ表示） のページに「誤検知の可能性が高いアラートをまとめて対応済みにする」ボタンがあります。
  • ランクで絞り込み：受信箱・効果大幅変動・新規未表示・利用率異常 のページで「すべて／緊急／要確認／誤検知」で絞り込めます。
  • ランク1（緊急）は1件ずつ：まとめ処理は避け、内容を確認してから対応します。

第1部　基本機能

広告アカウント連携（対応媒体）

以下の媒体の広告アカウントを連携して、配信状況を取り込みます。

  • Google 広告
  • Meta 広告（Facebook / Instagram）
  • LINEヤフー広告 ― 検索広告（リスティング）とディスプレイ広告（YDA）は別々のアカウント・別々の連携として扱います
  • TikTok 広告

補助的に、決済エラー検知のための Gmail 監視、通知先としての Slack 連携も利用できます。

配信計画管理（3階層）

キャンペーン／広告グループ（広告セット）／広告 の3階層で、AdAlert 上に 配信の開始・停止予定（計画） を持てます。媒体側では広告単位の細かい期間設定ができないため、AdAlert が計画を「正」として管理し、計画と実際の配信のズレをアラートで知らせます。上位階層で期間を設定すると配下にも反映できます。

データの取り込み（1時間ごと）

各媒体から 1時間ごと に当日の累積の数値（広告表示・利用額など）を取り込み、前回との差から「その1時間ぶんの実績」を計算して記録します。集計は世界標準時（UTC）の1時間区切りです。この取り込み結果をもとに、各種アラートの判定が動きます。

案件（プロジェクト）・予算管理

複数の広告アカウントを媒体をまたいで1つの 案件 にまとめ、fee率（手数料率）と予算を設定できます。予算の持ち方は2種類です。

  • 月予算（毎月の予算）
  • 期間予算（開始～終了の期間に対する予算。途中の予算改定にも対応）

この予算に対して着地予測（このままいくと使い切るか／余るか）を見て「利用率異常」を判定します。

通知

アラートは メール ／ Slack ／ Teams（Power Automate 経由）で通知できます。通知の 対象範囲（スコープ） を、会社全体／広告アカウント単位／キャンペーン単位／案件単位 のいずれかで絞り込めます。「この案件のアラートだけこの担当者へ」といった振り分けが可能です。

プライム代理店への共有（監視専門の代理店）

監視専門の代理店（プライム代理店）に、広告アカウント単位でデータを共有できます。プライム代理店は 閲覧のみ で、広告の直接編集はできません。共有したアカウント配下に後から追加された広告も自動的に共有対象になります。

変更依頼のワークフロー

配信スケジュールや監視設定の変更を、依頼 → 承認 → 反映 の流れで管理します。

  • 依頼できる内容：スケジュール変更／監視設定の変更
  • 依頼者：通常代理店 または プライム代理店（プライムは依頼のみ、承認は通常代理店側）
  • 状態：受付中 → 完了／却下／失敗

第2部　検知アラートの仕様

各アラートを同じ項目立てで記載します。「どういう時に出るか」「出たらどうするか」を中心に読んでください。

配信中のはずの広告で広告表示（インプレッション）が出ていない＝「開始したのに表示されていない」反映漏れを見つけます。

どういう時に出るか：直近24時間の実測データでコンバージョン合計が0のとき。ただし 24時間分がきちんと実測データで揃っている場合のみ 判定します（計測開始直後は判定しません）。

何と比べる

通知と重複

自動では消えません。

気をつけたい点そもそもコンバージョンが発生しにくい商材・時間帯では鳴りやすくなります。CV監視をオンにする広告は絞るのが有効です。

利用率異常（予算着地予測）

案件（プロジェクト）の予算に対し、このままの消化ペースだと着地が目標範囲を外れそうなときに知らせます。「使い切れない／使いすぎ」を事前に察知します。

重要度
重大
出る単位
案件
チェック
1日1回（深夜）
どういう時に出るか：現在までの消化額＋今後の想定（配信中キャンペーンの日予算合計 × 残日数）から着地額を見積もり、fee率を加味して「着地利用率」を算出。目標範囲（初期設定は 97〜99%）を外れると検知します。
判定の目安
90%未満＝大幅未達／90〜97%＝未達／97〜99%＝正常／99〜110%＝超過／110%超＝大幅超過。この範囲は案件ごと・会社ごとに変更できます。
何と比べる
案件の予算（月予算 または 期間予算）と、消化ペースからの着地予測。
対象になる／ならない
予算が設定され、配信中キャンペーンがある案件。予算未設定・0円、期間外の日、fee率が不正な案件は対象外。
媒体の注意
媒体を問わず案件配下の全アカウントを合算。円以外のアカウントは円換算して合算し、換算できなかった分は合計から除くため、着地が実態より低めに出ることがあります。
通知と重複
1日1回。対応済みにしても、翌日また外れていれば再び未対応として上がってきます。
解決の扱い
自動では消えません（正常域に戻っても手動で対応済みにするまで残ります）。
出たらどうする？未達なら予算増額・配信拡大を、超過なら日予算調整・停止を検討します。
気をつけたい点「日予算 × 残日数」の単純な見積もりのため、途中で大きな予算変更や配信ムラがあると予測がぶれます。

第3部　事前アラート（配信開始/停止の予定通知）

異常の検知とは別に、配信の開始予定・停止予定が近づいたら事前に知らせる 通知です。「開始し忘れ」「止め忘れ」を未然に防ぎます。

  • これは何？：AdAlert 上で設定した配信開始/停止の予定時刻が近づくと、事前に通知します。
  • いつ通知される？：予定時刻の 10分前 が初期設定です（会社ごとに変更可能）。システムは5分ごとに予定をチェックしています。
  • 対象：広告単位の配信予定。AdAlert 上で配信予定が設定され、監視がオンになっている広告。
  • 開始／停止の通知：開始予定・停止予定それぞれの時刻が近づいたとき。あわせて、設定した「配信停止期間」の開始前・終了前も通知できます。
  • 送信の確実さ：全チャネルで失敗した場合は「送信済み」にせず、次のチェックで再送を試みます。

第4部　アラート共通の挙動

同じ広告で何度も鳴る？（重複の抑制）

短時間に同じアラートが何度も飛ばないように抑制しています。

  • 広告表示・利用額など時間単位で見るアラート … 同じ時間帯（1時間のまとまり）につき1回 まで。
  • 利用率異常・決済エラー・CV未発生 … 1日につき1回 まで。

複数の監視処理が同時に走っても二重送信されないよう、送信担当を1つに絞る仕組みが入っています。

アラートの解決（対応済みの扱い）

検知されたアラートは 自動では消えません。担当者が内容を確認し、手動で「対応済み」にするまで残り続けます（見落とし防止のため）。

  • 翌日以降も同じ状態が続く場合、同じアラートとして扱われ、再通知は上のルールに従います。
  • 例外：広告審査落ち だけは、広告が再び「承認」状態に戻ると自動的に解決済みになります。
  • まとめて対応済み（ランク3）：誤検知の可能性が高いアラートは、受信箱や対象ページから一斉にクローズできます。詳しくは ランクの見方（対応の優先度） を参照してください。
